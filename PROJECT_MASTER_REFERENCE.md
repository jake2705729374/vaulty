## 1. What This App Is

**Vaultly** is a private, encrypted personal journal web app built with Next.js. It is inspired by Penzu and is designed primarily for mobile (iPhone browser) but works on desktop too.

The core promise: **your journal entries are encrypted on your device before they ever leave it.** The server stores only ciphertext — it is architecturally impossible for the server, the database, or the developer to read your entries.

On top of the private journal core, Vaultly layers several AI-powered features:

- **AI Coach** — a live, personalized chat assistant embedded directly in the journal editor that knows your people, your current life situations, and (optionally) the content of your recent entries
- **Post-entry Reflection** — an automatic short reflection generated after you save an entry
- **Mood Tracking & Insights** — per-entry mood logging with a 7-day strip, full calendar view, and AI-generated trend insights
- **Habits Tracker** — a simple daily habit tracker with streaks, completion rings, and AI habit insights
- **Weekly Digest** — an opt-in weekly email summary of your journaling week, generated by Claude
- **Memories** — the coach automatically extracts durable facts from your conversations and saves them so it remembers key details across sessions

---

## 2. Who It's For

Vaultly is a single-user, personal app. There is no social component, no sharing, no public content. It is for people who:

- Want a private, secure place to journal
- Want AI support that actually knows them (their relationships, their situations, their goals)
- Care deeply about privacy — nothing in plaintext on the server, ever
- Use their iPhone as their primary device

## 4. High-Level Architecture

```
Browser (Client)
  │
  ├── Encryption layer (lib/crypto.ts) — Web Crypto API
  │     Entry plaintext never leaves this layer
  │
  ├── Next.js App Router pages
  │     /journal, /coach, /habits, /settings, etc.
  │
  └── fetch() calls to:
        │
        ├── /api/* (Node Runtime on Vercel)
        │     ├── Prisma → PostgreSQL (Supabase)
        │     ├── Anthropic SDK → Claude API
        │     ├── Resend → email
        │     └── Supabase Storage REST API → object storage
        │
        └── /api/auth/* (NextAuth)
              └── Credentials provider (bcrypt)

Edge Runtime (proxy.ts)
  - Runs BEFORE every request to protected routes
  - Validates JWT, redirects unauthenticated users to /login
  - Adds security headers (CSP, HSTS, X-Frame-Options, etc.)
  - NO Prisma, NO Credentials provider (Edge-incompatible)

Observability
  - Sentry captures exceptions (client + server + edge)
  - Vercel Analytics tracks custom events (lib/analytics.ts)
  - Vercel Speed Insights captures Web Vitals

Cron Jobs (Vercel)
  - Daily 09:00 UTC → GET /api/digest/weekly (processes all opted-in users)
```

**Critical separation:**
- `proxy.ts` + `auth.config.ts` → Edge Runtime (no Node APIs, no Prisma)
- `lib/auth.ts` → Node Runtime (Credentials provider, bcrypt, Prisma)
- All entry encryption/decryption → Client-side only (Web Crypto API)

## 5. Security & Encryption Architecture

### 5.1 Two-Layer Envelope Encryption

```
User Password
    │
    ▼ PBKDF2 (100,000 iterations, 16-byte random salt, SHA-256)
Key Encryption Key (KEK)  ──────────────────────────────► stored: kekSalt
    │
    ▼ AES-256-GCM wrap
Master Encryption Key (MEK)  ──────────────────────────► stored: encryptedMek + mekIv
    │
    ▼ AES-256-GCM encrypt (per entry, random IV)
Ciphertext  ────────────────────────────────────────────► stored: ciphertext + iv
```

**Why envelope encryption?**
If you used direct password-based encryption on every entry, changing your password would require re-encrypting every entry. With envelope encryption:
- Password change = re-wrap the MEK with a new KEK. All entries remain untouched.
- The MEK is derived once per session and held in memory (never stored in plaintext anywhere).

### 5.2 Key Derivation (lib/crypto.ts)

| Function | Purpose |
|---|---|
| `generateSalt()` | Random 16-byte salt for PBKDF2 |
| `deriveKey(password, salt)` | PBKDF2 → AES-256-GCM CryptoKey |
| `deriveKek(password, salt)` | Alias for deriveKey (clarity) |
| `generateMek()` | Random 256-bit AES-256-GCM CryptoKey |
| `encryptMek(mek, kek)` | Wrap MEK: returns { encryptedMek (base64), mekIv (base64) } |
| `decryptMek(encryptedMek, mekIv, kek)` | Unwrap MEK: returns CryptoKey |
| `unlockMek(password, bundle)` | Full unlock: password + DB bundle → CryptoKey |
| `createKeyBundle(password)` | Registration: generate MEK + wrap it → bundle for DB |
| `rewrapMek(mek, newPassword)` | Password change: re-wrap existing MEK |
| `encryptWithMek(plaintext, mek)` | Encrypt entry: returns { ciphertext, iv } (no salt needed) |
| `decryptWithMek(ciphertext, iv, mek)` | Decrypt entry |
| `encryptEntry(plaintext, password)` | Legacy: per-entry PBKDF2. Returns { ciphertext, iv, salt } |
| `decryptEntry(ciphertext, iv, salt, password)` | Legacy decryption |

### 5.3 Legacy Migration

Older entries use per-entry PBKDF2 key derivation (the "legacy" path) with a `salt` field.
New entries use MEK encryption with `salt = null`.

On first login, if the user has no key bundle stored, the app runs a one-time migration:
1. Derive KEK from master password
2. Generate a new MEK
3. POST `/api/user/migrate-keys` with the encrypted MEK bundle + all re-encrypted entries
4. Server atomically upserts the key bundle and updates all entries

### 5.4 Master Password Session

- User enters master password at the `LockScreen` component
- Derived MEK stored in component state (React memory only)
- `sessionStorage.getItem("masterPassword")` is used to persist across page navigations within the same browser tab session
- On sign-out: `sessionStorage.removeItem("masterPassword")` clears it immediately

### 5.5 What the Server NEVER Sees

- Entry plaintext (always ciphertext in DB)
- The MEK (only the encrypted form is stored)
- The user's master password (only bcrypt hash of account password; MEK derivation is client-only)

### 5.6 Edge Middleware Security Headers

Every response through `proxy.ts` gets:
- `X-Frame-Options: DENY` — blocks iframe embedding (clickjacking)
- `X-Content-Type-Options: nosniff` — blocks MIME-type sniffing
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: camera=(), microphone=(), geolocation=()`
- `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`

## 6. Authentication System

### Stack: NextAuth.js v5 + Credentials Provider

**Files:**
- `auth.config.ts` — Edge-compatible config (JWT + route protection, no Node imports)
- `lib/auth.ts` — Full NextAuth config (Credentials provider, bcrypt, Prisma)
- `proxy.ts` — Edge middleware using `auth.config.ts` only

### Sign-Up Flow
1. `POST /api/auth/register` — hash password with bcrypt, create User, send verification email via Resend
2. User receives 6-digit OTP email → `POST /api/auth/verify-email` — sets `emailVerified = true`
3. User can now log in

### Sign-In Flow
1. NextAuth Credentials `authorize()`:
   - Rate-limit check (Upstash, by IP)
   - Find user by email
   - `bcrypt.compare(password, passwordHash)`
   - Check `emailVerified` — throws if unverified
   - Returns `{ id, email }`
2. JWT created with `token.id = user.id`
3. Session callback: `session.user.id = token.id`

### Password Reset Flow
1. `POST /api/auth/forgot-password` — generate 6-digit OTP, SHA-256 hash stored in DB, send OTP email (15-min TTL)
2. `POST /api/auth/reset-password` — verify OTP, bcrypt new password, re-wrap MEK with new KEK, update DB

### Sign-Out
- Settings button → `sessionStorage.removeItem("masterPassword")` then `window.location.href = "/api/auth/logout"`
- `GET /api/auth/logout` (Node runtime, NOT Edge) — deletes all NextAuth session/CSRF/callback cookies, redirects to `/`
- This approach bypasses the Edge middleware entirely (sign-out does not hit `proxy.ts`)

### Protected Routes (gated by proxy.ts)
```
/dashboard, /journal, /coach, /habits, /therapist, /settings, /onboarding
```
Unauthenticated requests are redirected to `/login` by NextAuth's `authorized` callback.

### 8.4 AI Coach (In-Editor Panel)

**File:** `components/CoachPanel.tsx`
**API:** `POST /api/coach/chat`

The coach is the most personalized AI feature in the app. It knows the user's name, their key relationships, their current life situations, and — optionally — the content of their recent journal entries.

#### What the Coach Knows

The server assembles a dynamic context block for every message:

| Context | Source | Always Sent? |
|---|---|---|
| User's display name | UserPreferences | Yes (if set) |
| Coaching style preference | UserPreferences.coachStyle | Yes (if not "balanced") |
| User bio | UserPreferences.userBio | Yes (if set) |
| Key people (name, relationship, birthday, traits, notes) | UserPreferences.coachPeople | Yes (if any) |
| Birthday alerts (within 7 days) | Computed from coachPeople | Yes (if relevant) |
| Life phase | UserPreferences.coachLifeContext.phase | Yes (if set) |
| Current situations | UserPreferences.coachLifeContext.situations | Yes (if any) |
| Memories (past insights) | Memory table, last 10 | Yes (from DB) |
| Session summaries | SessionSummary table, last 5 | Yes (from DB) |
| Current entry text (live) | Props from editor | Yes (if non-empty) |
| Recent entries (last 5) | Decrypted client-side, passed in body | Only if privacy toggle ON |
| Today's habits status | Habit + HabitLog tables | Yes (from DB, fetched server-side) |
| Safety flag | Pre-check result from /api/coach/safety | Only if triggered |

#### Coaching Styles

| Style | Behavior |
|---|---|
| `gentle` | Extra validating, prioritizes comfort |
| `direct` | Concise, honest, skips softening |
| `challenging` | Pushes for growth, questions assumptions |
| `balanced` (default) | Warm but practical |

#### Starter Prompts

When the panel is first opened (no messages yet), up to 4 suggestion chips appear:
- `"How are things with [Person]?"` — one per person (up to 2)
- Situation-based prompt (e.g., if "Breakup" situation: "How are you processing things emotionally right now?")
- `"What was the highlight of your day?"`
- `"How are you feeling right now?"`

Clicking a chip instantly sends the message.

#### Add to Entry

Below each completed assistant message, an "Add to Entry" button appends the coach's response (markdown stripped) to the bottom of the journal editor. Not shown while message is still streaming.

#### Memory Extraction (fire-and-forget)

After each coach response, `POST /api/coach/extract` runs asynchronously. Claude reads the conversation and extracts durable facts (e.g., "User's sister Emma is visiting next month"), saving them as Memory records for future sessions.

#### Session Summaries

After each conversation, `POST /api/session-summary` creates a 1–2 sentence summary of what was covered. This is injected into future coach sessions so the coach can pick up where it left off without re-reading full history.

#### Safety Check

`POST /api/coach/safety` runs a pre-check on the user's message before sending to the main coach. If elevated concern is detected (distress signals), the `elevated` flag is set to `true` in the coach payload, which causes the coach to respond with extra warmth and gently encourage professional support.

#### No-Profile Nudge

If `coachContext` is null (user hasn't set up their coach profile), a banner appears: `"Set up your Coach Profile in Settings →"` linking to `/settings#coach-profile`.

function localIso(): string {

### 8.8 Weekly Digest

**File:** `app/api/digest/weekly/route.ts`

An opt-in feature that emails the user a warm weekly summary of their journaling activity.

**Two callers:**
1. **Vercel Cron** (daily 09:00 UTC, configured in `vercel.json`) — processes all opted-in users. Authenticated via `Authorization: Bearer <CRON_SECRET>` header.
2. **Client dashboard load** — processes only the current user via NextAuth session. No-ops if last digest was < 6 days ago.

**Logic:**
1. Check `digestEnabled = true` and `lastDigestAt` is more than 6 days ago (prevents double-sends)
2. Fetch entry titles + moods from the last 7 days (titles are plaintext — body ciphertext is NOT used)
3. Fetch saved memories + session summaries from the past week
4. Build plain-text digest input for Claude (entry date + title + mood, memories, session notes)
5. Generate digest via Claude (DIGEST_SYSTEM_PROMPT, max 600 tokens)
6. Send email via Resend to the user's registered email
7. Update `lastDigestAt` in DB

**Privacy note:** The digest uses only entry **titles** and **moods** — not the encrypted body. The server never sends ciphertext to Claude. This is consistent with the "server never sees plaintext" architecture because titles are stored as plaintext (intentional design choice).

### 8.12 Media Attachments

**Storage:** Supabase Storage (`entry-media` bucket, public)
**API:** `app/api/entries/[id]/media/`
**Library:** `lib/storage.ts`

Photos and videos attached to journal entries are stored in Supabase Storage object storage — not in Postgres. Only the CDN URL is kept in the database.

#### Upload Flow
1. User selects file in JournalEditor
2. Client reads file as `ArrayBuffer` via `file.arrayBuffer()`
3. `POST /api/entries/[id]/media` — multipart form data
4. Server creates DB record to get stable `mediaId` (cuid)
5. Server calls `uploadMedia(arrayBuffer, mimeType, fileName, userId, mediaId)` in `lib/storage.ts`
6. `lib/storage.ts` calls Supabase Storage REST API: `POST /storage/v1/object/entry-media/{userId}/{mediaId}-{safeName}`
7. Returns public CDN URL
8. Server updates `EntryMedia.storageUrl` with CDN URL
9. Client displays thumbnail using `blobUrl` (instant preview) then `storageUrl` on reload

#### Delete Flow
1. `DELETE /api/entries/[id]/media/[mediaId]`
2. Server calls `deleteMedia(storageUrl)` — Supabase Storage REST API `DELETE` with `{ prefixes: [path] }`
3. Server deletes `EntryMedia` DB record
4. Storage errors are non-fatal (stale URLs acceptable; 404s silently succeed)

#### Storage Path Format
```
entry-media/{userId}/{mediaId}-{sanitizedFileName}
```
Non-guessable paths (cuid IDs) provide security through obscurity. Public bucket — no signed URLs needed.

#### `lib/storage.ts` Functions
```typescript
uploadMedia(fileBuffer: ArrayBuffer, mimeType: string, fileName: string, userId: string, mediaId: string): Promise<string>
deleteMedia(storageUrl: string): Promise<void>
```

**Required env vars:** `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`

## 9. AI Integration

### Model

All AI features use **claude-haiku-4-5-20251001** (Claude Haiku). This is a fast, cost-efficient model appropriate for the response lengths and latency requirements of a mobile journaling app.

### Anthropic Client

```typescript
// lib/ai/claude.ts
import Anthropic from "@anthropic-ai/sdk"
export const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })
```

### Prompt Caching

All large system prompts use `cache_control: { type: "ephemeral" }` on the first message block. This tells the Anthropic API to cache the prompt for up to 5 minutes, dramatically reducing cost for repeated calls (e.g., every coach message in a conversation reuses the same cached base prompt).

```typescript
const systemBlocks = [
  {
    type: "text",
    text: COACH_BASE_PROMPT,
    cache_control: { type: "ephemeral" },  // cached — same for all users
  },
  {
    type: "text",
    text: contextBlock,
    // No cache_control — changes per user, must not be cached
  },
]
```

### All System Prompts (`lib/ai/prompts.ts`)

| Prompt | Used In | Tone/Behavior |
|---|---|---|
| `COACH_BASE_PROMPT` | `/api/coach/chat` | Warm life coach. 2–4 sentences + 1 focused question. Not a therapist. Celebrates wins. References people by name. Handles journaling requests (drafts in first person). Crisis: redirects to 988 Lifeline. |
| `THERAPIST_SYSTEM_PROMPT` | `/api/therapist/chat` | Empathetic journaling companion. Open-ended questions. Never clinical. Never substitutes for professional care. Crisis redirect. |
| `REFLECTION_SYSTEM_PROMPT` | `/api/therapist/reflect` | 3–5 sentences. Notices 1–2 themes/emotions. Closes with a gentle question. Second-person. |
| `SESSION_SUMMARY_PROMPT` | `/api/session-summary` | 1–2 sentences. Third-person ("the user"). Captures themes/emotions/decisions. No preamble. |
| `DIGEST_SYSTEM_PROMPT` | `/api/digest/weekly` | Personal warm letter. 3–5 sentences. Acknowledges week's tone, mood patterns, meaningful element, encouraging close. |
| `INSIGHTS_SYSTEM_PROMPT` | `/api/therapist/insights` | 4–6 sentences. Mood trend, frequent mood, one observation, one suggestion. Honest but kind, no diagnosis. |

### Streaming Pattern

All chat endpoints return a streaming response:

```typescript
// Server (API route)
const stream = anthropic.messages.stream({ ... })
const readable = new ReadableStream({
  async start(controller) {
    stream.on("text", (text) => controller.enqueue(encoder.encode(text)))
    await stream.finalMessage()
    controller.close()
  },
  cancel() { stream.abort() },
})
return new Response(readable, {
  headers: { "Content-Type": "text/plain; charset=utf-8", "Cache-Control": "no-cache" }
})

// Client (component)
const reader = response.body.getReader()
const decoder = new TextDecoder()
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  setCurrentMessage(prev => prev + decoder.decode(value))
}
```

## 12. Key Workflows (Step-by-Step)

### New User Registration → First Entry

```
1. User visits /register
2. Enters email + password → POST /api/auth/register
   - Bcrypt password hash, create User record, send verification email (Resend)
3. Clicks 6-digit OTP → POST /api/auth/verify-email
   - emailVerified = true
4. Logs in → POST /api/auth/[...nextauth] (Credentials)
   - Rate limit check (Upstash), bcrypt compare, emailVerified check
   - JWT created, session stored
5. proxy.ts validates JWT on every protected route request
6. Redirected to /onboarding (onboardingDone = false)
7. Completes 8-step onboarding → PATCH /api/user/preferences + POST /api/habits
8. Redirected to /dashboard
9. Clicks "Write Entry" → /journal
10. LockScreen prompts for master password
11. GET /api/user/key-bundle → no bundle found
12. One-time migration: generateMek() + createKeyBundle() → POST /api/user/migrate-keys
    - Key bundle stored in DB; all entries (empty at this point) migrated
13. JournalEditor opens (unlocked)
14. User writes → encryptWithMek() client-side → POST /api/entries
15. Auto-save every 2s → PUT /api/entries/[id]
16. Coach panel opened → POST /api/coach/chat (streaming)
17. Sentry + Vercel Analytics active throughout for error/event tracking
```

### Returning User Opening an Entry

```
1. /journal loads, LockScreen prompts
2. Master password entered → GET /api/user/key-bundle
3. unlockMek(password, bundle) → MEK in memory
4. GET /api/entries → list renders (titles only, no decryption needed)
5. User clicks entry → /journal/[id]
6. GET /api/entries/[id] → { ciphertext, iv, salt, media: [{storageUrl, ...}] }
7. salt === null → decryptWithMek(ciphertext, iv, mek)
   salt !== null → decryptEntry(ciphertext, iv, salt, password)  [legacy]
8. Editor populated with decrypted content
9. Media thumbnails load from storageUrl (Supabase Storage CDN)
```

### Password Change

```
1. Settings → Password section
2. User enters current + new password
3. POST /api/user/password
   - bcrypt.compare(currentPassword, passwordHash) — fail fast if wrong
   - bcrypt.hash(newPassword) — new account password hash
   - deriveKek(newPassword, newSalt) — new KEK
   - rewrapMek(existingMek, newPassword) — re-wrap with new KEK
   - Update User.passwordHash + User.encryptedMek + User.mekIv + User.kekSalt atomically
4. All entries remain unchanged (MEK unchanged, only its wrapping changes)
```

### Coach Conversation (In-Editor)

```
1. User opens JournalEditor, writes content (live text fed via onEditorTextChange debounced 500ms)
2. Clicks Coach button → CoachPanel opens
3. If entryId exists: GET /api/entries/[id]/chat → load history
4. Starter prompts shown (built from coachContext.people + situations)
5. User clicks prompt or types message
6. POST /api/coach/safety → check for elevated distress
7. POST /api/coach/chat → {message, history, coachContext, entryContent, recentEntries?, elevated}
   Server side:
   - Fetch memories (last 10), session summaries (last 5), prefs (bio, style), habits (today)
   - Build contextBlock (dynamic, not cached)
   - anthropic.messages.stream({model, system: [cachedBasePrompt, contextBlock], messages})
   - Stream ReadableStream back to client
8. Client reads stream, updates assistant message character by character
9. Message complete → "Add to Entry" button appears
10. Fire-and-forget: POST /api/coach/extract (extract facts → Memory)
11. POST /api/entries/[id]/chat (persist messages to DB)
12. POST /api/session-summary (summarize session → SessionSummary)
13. track("coach_message_sent") analytics event fired
```

### Coach Session Auto-Title (Standalone Coach)

```
1. User sends 4th message in a coach session (2nd full exchange)
2. Client detects newMessages.length + 1 === 4
3. Fire-and-forget: POST /api/coach/sessions/[id]/generate-title
   Server side:
   - Load first 6 messages
   - Build compact transcript (role: content.slice(0, 200))
   - claude-haiku: "4–7 words, specific, sentence case, no quotes, reply with title only"
   - Strip quotes, limit to 100 chars
   - Update CoachSession.title
4. Client optimistically updates sidebar: setSessions(prev => prev.map(...))
```

### Media Upload

```
1. User clicks attach (or drag-drop) in JournalEditor
2. file.arrayBuffer() → ArrayBuffer
3. POST /api/entries/[id]/media (multipart form data)
   Server:
   - Create EntryMedia DB record (gets stable mediaId)
   - uploadMedia(arrayBuffer, mimeType, fileName, userId, mediaId)
     → Supabase Storage REST API: POST /storage/v1/object/entry-media/{userId}/{mediaId}-{name}
     → Returns public CDN URL
   - Update EntryMedia.storageUrl
   - If storage fails: delete the DB record, return 500
4. Client receives { storageUrl, ... }
5. Displays thumbnail via blobUrl (instant) then storageUrl on next load
6. track("media_uploaded", { media_type }) analytics event
```

## 15. Known Architectural Decisions & Constraints

### Must Know Before Touching Code

1. **`proxy.ts` not `middleware.ts`** — Next.js 16 deprecated `middleware.ts`. The Edge middleware file MUST be named `proxy.ts`. Using `middleware.ts` will fail with a build error.

2. **`auth.config.ts` is Edge-only** — This file is imported by `proxy.ts` (Edge Runtime). It must NEVER import Prisma, bcrypt, `next-auth/providers/credentials`, or any Node.js built-in. Those imports crash the Edge Runtime with `MIDDLEWARE_INVOCATION_FAILED`. The Credentials provider lives exclusively in `lib/auth.ts`.

3. **Sign-out via `/api/auth/logout` only** — Never use `signOut()` from `next-auth/react`. That client-side function creates a redirect that hits the Edge middleware while the JWT cookie is in a transitioning state, causing `jwtDecrypt` to throw `AggregateError` and crash the Edge Runtime. The sign-out button must navigate to `/api/auth/logout` (Node runtime, not in middleware matcher).

4. **Entry titles are plaintext** — This is intentional for performance (list views don't require decryption). Users should be aware their titles are visible to the server. The entry body is always encrypted.

5. **`salt = null` means MEK-encrypted** — When an entry has `salt = null`, it was encrypted with `encryptWithMek`. When `salt` is a string, it was encrypted with the legacy `encryptEntry` (per-entry PBKDF2). Always check before decrypting.

6. **Habit dates use client local time** — `new Date().toISOString()` is NOT used for habit logs because it returns UTC. The `localIso()` helper constructs the date string using local timezone. This is critical — UTC midnight vs local midnight is different and can cause habits to appear on the wrong calendar day.

7. **Coach context is per-user, not cached** — The dynamic context block (name, people, situations, entry text, habits) is assembled fresh on every API call and NOT marked with `cache_control`. Only the static `COACH_BASE_PROMPT` is cached. Mixing these up would leak user data between cached responses.

8. **Recent entries for coach are decrypted client-side** — When `coachContextEnabled = true`, the browser fetches ciphertext from `/api/entries/recent`, decrypts locally, and sends plaintext to `/api/coach/chat`. The server API then sends that plaintext to Claude. This is disclosed to the user via the privacy modal in Settings. The server never independently stores or logs these decrypted texts.

9. **`export const proxy = auth(callback)` pattern** — The NextAuth v5 middleware must use the callback form: `auth((_req) => { ... })`. Direct call like `auth as (req) => Promise<Response>` fails TypeScript type checks due to NextAuth's overloaded type signatures.

10. **Vercel auto-deploy owns the production alias** — If you run `vercel --prod --yes` manually, you'll get a different alias than the GitHub integration deploys to. Push to `main` via Git to deploy to the canonical production URL.

11. **Never read `EntryMedia.data`** — The `data Bytes` column was dropped in migration `20260429000000_media_object_storage`. Media binary data now lives in Supabase Storage. Always use `storageUrl`. Accessing `data` will throw a Prisma error.

12. **Always use `parseBody()` for API routes that accept JSON** — `lib/validation.ts` exports `parseBody(rawText, schema, maxBytes)` which handles byte-size limits, JSON parsing errors, and Zod validation in one call, returning a ready-made `NextResponse` on failure. Every API route that accepts a JSON body must use this — do not roll your own parsing.

13. **Vercel Cron auth is `Authorization: Bearer`** — The digest cron uses the standard `Authorization: Bearer <CRON_SECRET>` header, not a custom `x-cron-secret` header. Set `CRON_SECRET` in Vercel environment variables. Use `printf` (not `echo`) when setting via CLI to avoid trailing newline.

14. **Schema changes: always use `migrate`, never `db push`** — `npx prisma db push` bypasses migration history. Always use `npx prisma migrate dev --name <description>` for dev and `npx prisma migrate deploy` for production.

15. **Media upload uses `ArrayBuffer`, not `Buffer`** — `lib/storage.ts` accepts `fileBuffer: ArrayBuffer` (not Node `Buffer`) to satisfy TypeScript's `BodyInit` type for the fetch API. Call `file.arrayBuffer()` directly in the route handler and pass the result.

## 16. Deployment & CI/CD

### CI Pipeline (GitHub Actions)

**File:** `.github/workflows/ci.yml`

Two jobs run on every push and PR to `main`:

| Job | Trigger | Steps |
|---|---|---|
| `check` | Every push + every PR to `main` | `npm ci` → ESLint lint → `npm test` (Vitest) → `npm run build` |
| `deploy` | Push to `main` only (after `check` passes) | `vercel deploy --prod` |

CI uses placeholder env vars so `next build` compiles without a real DB connection. The `VERCEL_TOKEN` secret must be set in GitHub repo Settings → Secrets.

### Running Tests

```bash
npm test          # Run all tests once (Vitest)
npm run test:watch  # Watch mode
```

Test files: `__tests__/lib/crypto.test.ts` (40 tests) and `__tests__/lib/password-strength.test.ts` (18 tests).

### Production Environment
- **Hosting:** Vercel
- **Production URL:** `vaultly-sepia.vercel.app`
- **Database:** PostgreSQL (Supabase)
- **Branch:** `main` → auto-deploys on push via GitHub Actions

### Deploy Process
```bash
git push origin main    # Triggers GitHub Actions → Vercel production deploy
```

### After Any Backend Schema Change
```bash
npx prisma migrate dev --name <description>   # Create + apply migration (dev)
npx prisma migrate deploy                     # Apply to production
npm run build                                 # Verify TypeScript compiles (prisma generate is part of build)
git push origin main                          # Deploy
```

### Build Command
```json
"build": "prisma generate && next build"
```
Prisma generates the client first, then Next.js builds. Source maps are uploaded to Sentry at build time and deleted from the deploy output.

### Backup
- Raspberry Pi runs shell scripts + cron jobs to back up the PostgreSQL database
- No Node.js on the Pi — shell-only scripts
- Separate from Vercel deployment; Pi has its own SSH access to the DB

### Cron Jobs
- **Weekly Digest:** Configured in `vercel.json` — fires daily at 09:00 UTC, hits `GET /api/digest/weekly` with `Authorization: Bearer <CRON_SECRET>`
